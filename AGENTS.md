# AGENTS.md

Authoritative guide for AI coding agents working in this repository. Read this file in full before making any change. If a request would conflict with anything documented here, **update this file as part of the same change** so it stays the source of truth.

## What this project is

A locally-run Streamlit desktop app that turns any model served by the [W&B Inference](https://docs.wandb.ai/inference) service into a tool-calling code editing agent. The user supplies their W&B API key, picks a model from the live `/v1/models` list, points the agent at a working directory on disk, and chats with it. The agent autonomously calls a small set of tools (`list_files`, `read_file`, `write_file`, `edit_file`, `run_shell`) to read and modify code, with arguments and unified diffs shown inline in the chat for full transparency. The user is responsible for choosing a working directory whose contents they're comfortable letting the agent run shell commands inside; there is no in-product gate on filesystem writes or shell execution beyond the working-directory sandbox.

Every chat-completion call to the W&B Inference service, every agent turn, and every tool dispatch is traced to [W&B Weave](https://docs.wandb.ai/weave/) — initialized in the connect flow with the same API key and `team/project` the user pastes for inference attribution — so the user gets a structured timeline of inputs, outputs, latency, and token usage in their W&B workspace without any extra setup.

The app ships in two equivalent forms: directly via `streamlit run streamlit_app.py` for development, and as a packaged desktop application built by [`scripts/build_desktop.py`](scripts/build_desktop.py) (a `.app` on macOS, an "onedir" launcher on Linux/Windows). Both render through the same `streamlit_app.py` entry point.

## Module map - read this before adding files

The runtime is split into nineteen Python modules:

- **Top-level Streamlit entry + UI helpers**: `streamlit_app.py`, `chat_input.py`, `theme_switcher.py`, `font_size_switcher.py`.
- **Per-page bodies under `app_pages/`**: `app_pages/chat.py` (the chat experience — also owns the live working-tree diff via a "Changes" button + `@st.dialog` modal mounted from the chat page), `app_pages/usage.py` (the token-and-cost dashboard), `app_pages/settings.py` (GitHub identity + theme switcher + W&B Inference connection), and `app_pages/docs.py` (the in-app user guide — the user-facing companion to this file). Wired together by `st.navigation` in `streamlit_app.py`.
- **Cross-page callbacks**: `actions.py` — recents helpers, folder picker, Connect / Disconnect / Forget callbacks, GitHub PAT verify / sign-out callbacks. **Page modules import from here, not from `streamlit_app`**, because Streamlit loads the entry script as `__main__` and a `from streamlit_app import …` from a sub-page re-runs `main()` and re-renders the sidebar (producing duplicate-widget-key errors).
- **Agent + tools**: `agent.py`, `tools.py`.
- **Service clients**: `wb_client.py`, `mcp_servers.py`.
- **Pure data + helpers (no Streamlit imports)**: `models.py` (model metadata + pricing), `account.py` (GitHub PAT auth, opt-in saved credentials), `usage.py` (token-usage capture, cost compute, on-disk log), `project_context.py`, `git_ops.py` (single boundary for `git` subprocess calls + remote-compare-URL building), `commit_ai.py` (DeepSeek-driven commit-message + PR-description generation for the chat-page push flow), `chats.py` (multi-chat data model, on-disk persistence, and per-chat background-thread turn runner).

Plus the build script (`scripts/build_desktop.py`) and the Streamlit config file. **Do not create new top-level modules without first re-reading this section and confirming the change does not duplicate existing functionality. If you add a module, document it here in the same edit.**

Hard rules:

- All chat-completion requests go out from **exactly one module**: `agent.py`. Two call sites live there: `_stream_one_call` (the streamed, tool-calling path under `run_agent_turn`) and `generate_text` (one-shot, non-streamed completions used by `commit_ai.py`'s commit-message + PR-description helpers, and any future AI text helper). Never call `client.chat.completions.create` from anywhere else.
- The OpenAI client is built **only** by `wb_client.make_client`.
- Every `git` invocation runs through `git_ops.py`. No other module shells out to `git` (with the narrow exception of `account.py`'s project-bootstrap helpers — `git_init`, `git_clone`, `git_add_remote`, `apply_git_identity` — which predate `git_ops.py` and own the GitHub-PAT-aware clone flow). Branch switching, status, diff, stage, commit, fetch, pull, push, conflict detection, and platform compare-URL building all live in `git_ops.py`.
- Filesystem reads and writes against the user's working directory live **only** in `tools.py`. Read-only scans for guidance files (`AGENTS.md`, `.cursor/rules`, and SKILL.md files under either `.cursor/skills` or `.claude/skills`) live **only** in `project_context.py`.
- The path-containment check (`_resolve_inside`) is the single safety boundary for filesystem tools. Any new file-touching tool **must** call it before reading or writing.
- W&B Weave is initialized **only** by `wb_client.init_weave`, called once per Connect from `streamlit_app.py`. Anywhere else that needs Weave should rely on the `@weave.op` decorator and the auto-patched OpenAI client; do not call `weave.init` from `agent.py`, `tools.py`, `mcp_servers.py`, or anywhere else.
- MCP servers are connected and dispatched **only** through `mcp_servers.MCPRegistry`. No other module opens MCP `ClientSession`s, parses MCP tool schemas, or writes `~/.wb_coding_agent/mcp.json`.
- Skill discovery + selection lives **only** in `project_context.py`. The agent loop calls `select_skills_for_turn(...)` once per turn and splices the returned text; nothing else mines `.cursor/skills` or `.claude/skills`. To add a new skill-directory convention, append to `SKILL_ROOT_SUBPATHS` in `project_context.py`; do not hard-code the path elsewhere.
- CCv2 components live in dedicated UI modules (today: only `chat_input.py`). Do not call `st.components.v2.component(...)` from `streamlit_app.py`; if you need a new component, add a new module beside `chat_input.py` and import its public mount function.

### Hard rules

These supersede the originals:

- **Outbound chat completions go through `chat_streams.stream_chat` only.** `agent._stream_one_call` is the single caller of `chat_streams.stream_chat`. Native SDKs (Anthropic / Google / OpenAI) are constructed via `providers.make_provider_client(...)` and stored in `ss.clients[<provider_id>]`; the 9 OpenAI-compat providers also use the OpenAI SDK (built by the same factory with a per-provider `base_url`) and stored alongside.
- **`model_catalog.py` is the only module that hits a provider's `/v1/models` endpoint or the OpenRouter `/api/v1/models` API.** Both caches under `~/.wb_coding_agent/` (live model-catalog snapshot + OpenRouter blob) are written only by this module.
- **Strict completeness gate**: a model with incomplete metadata (missing provider-accurate pricing, missing real description, missing context window for chat models) MUST NOT be returned from `available_qualified_ids` or rendered in the picker. The gate runs per-mode — chat needs both axis prices; image-gen needs `image_pricing`; etc. The dropped count is exposed via `hidden_models_summary(...)` and surfaced as a footer caption in the modal.
- **Pricing is per-(provider, model) tuple — never substituted across providers.** Pricing comes either from the curated `MODEL_METADATA` entry for that exact qualified id OR (for `openrouter:*` ids only) from OpenRouter's own `/api/v1/models` response. There is no third-party metadata scrape; there is no live-API pricing fallback for non-OpenRouter providers — we treat curated as the contract for everyone except OpenRouter where the pricing-and-billing relationship is direct. Models without curated pricing on non-OpenRouter providers get hidden by the strict gate — better silence than a wrong number on the Usage dashboard.
- **Description precedence is curated → provider's own `/v1/models` description → (for `openrouter:*` only) OpenRouter's catalog description.** The OpenRouter catalog never enriches descriptions for non-OpenRouter providers' models; that would silently substitute one provider's framing for another's offering.
- **Native SDKs are preferred for the five frontier providers** (OpenAI, Anthropic, Google, Mistral, xAI). Anthropic and Google use their native SDKs because the wire format is fundamentally different from OpenAI's. Mistral uses `mistralai.client.Mistral` because that's the official Python client and the dispatch path stays separate so future Mistral-only kwargs (FIM, structured outputs, Magistral reasoning options) can land in one branch without affecting other providers. xAI uses the OpenAI SDK with `base_url=https://api.x.ai/v1` (xAI's REST endpoint is officially OpenAI-compatible), but routes through a separate `xai_native` dispatch path so Live Search via `extra_body={"search_parameters": ...}` lands in one branch — the gRPC `xai-sdk` package is deliberately NOT pulled in because the REST surface covers everything we need and avoids the gRPC / OpenTelemetry dependency tail. OpenAI uses the OpenAI SDK natively because it owns the reference shape and is the source of OpenAI-only kwargs (`reasoning_effort` for o-series is wired today; typed `parse()` etc. are hooks-ready). The 2 OpenAI-compat providers (W&B Inference, OpenRouter) use the same OpenAI SDK with a per-provider `base_url` because each exposes an OpenAI-shape `/v1/chat/completions` endpoint with no app-relevant deviations.

- **Native feature wiring per provider** (active vs hooks-ready):
  - **OpenAI**: `reasoning_effort` (low / medium / high) — **active**; per-chat toggle on the chat page when an o-series model is selected. Typed `parse()`, `prediction`, Files / Batch / Vector Stores APIs — hooks-only.
  - **Anthropic**: prompt caching via `cache_control: ephemeral` on the system prompt — **active and unconditional** (auto-applied by `chat_streams._stream_anthropic_native`; no UI). Extended thinking, computer use / bash / text-editor pre-built tools, Message Batches API — hooks-only.
  - **Google Gemini**: Google Search grounding — **active**; per-chat toggle on the chat page when a Gemini model is selected. Auto-dropped when function tools are present (Gemini API can't combine the two). Code execution, `client.caches.create(...)` for context caching, native Imagen / Veo — hooks-only.
  - **Mistral**: chat completions (no native-only flags wired today). Mistral's native SDK exposes FIM, embeddings, classifiers, agents API, Magistral reasoning options — all hooks-ready via the dedicated `mistral_native` dispatch path.
  - **xAI**: Live Search (`mode="auto"`) — **active**; per-chat toggle on the chat page when a Grok model is selected. Reasoning effort, deferred completions, image generation — hooks-only.

- **Per-chat native-feature toggles flow through `Chat.model_options`.** The `Chat` dataclass carries a `model_options: dict[str, Any]` field (JSON-round-tripped) for per-chat feature flags. Recognized keys today: `reasoning_effort` (str: `"low"` / `"medium"` / `"high"`), `grounding` (bool), `live_search` (bool), `thinking` (bool, hook-only). The chat page renders the applicable toggle inline beneath the model card caption (only shown when the current model exposes the option). The toggle's `on_change` writes back to `chat.model_options` under `chat._lock` and calls `chats.save_chat`. The dict threads through `chats.start_turn` → `agent.run_agent_turn` → `agent._stream_one_call` → `chat_streams.stream_chat` to the per-kind dispatch path, which pulls the keys it cares about. Adding a new option is a 3-line change in `chat_streams.py` (key lookup + kwarg threading) plus a `_render_*_toggle` helper in `app_pages/chat.py`.
- **All media bytes (image / audio / video output AND uploaded attachment bytes) land inside `<working_dir>/.wb_artifacts/<chat_id>/` via `_artifact_paths`.** Generated outputs go through `tools.py`'s `generate_image / generate_speech / generate_video` tools (writing into `<chat_id>/`); user uploads go through `attachments.save_uploads` (writing into `<chat_id>/inbox/`). No other module writes media or attachment bytes.
- **Multimodal content array construction goes through `attachments.build_user_message` only.** `chat_streams.py`'s per-provider translators reshape the OpenAI-shape array into Anthropic / Google native shapes; they don't generate content arrays from raw uploads.
- **The chat-history media renderer in `app_pages/chat.py::_render_tool_event` is the only site that resolves a tool-result `kind` field (`"image"` / `"audio"` / `"video"`) into HTML5 elements via `st.image` / `st.audio` / `st.video`.** The user-turn renderer `_render_user_turn` is the parallel single site for inbound attachments.
- **The model picker UI is `app_pages/chat.py`'s `_model_picker_dialog` only.** Don't render an `st.selectbox` for models anywhere else; the modal is the single model-selection surface.
- **OpenRouter MUST render the `:gray-badge[marked-up gateway]` label and the markup-caveat caption.** OpenRouter is fundamentally a marked-up gateway (5–10% over native rates). It now sits in `tier == "primary"` (next to W&B + the 3 frontier providers) because elevated visibility makes the "one key for the long tail" trade-off explicit; the persistent label + caption are what keep users informed about what they're paying for. Don't drop the badge or the caption.

### Anti-duplication checklist (multi-provider additions)

- Don't build a second provider-client constructor — extend `providers.make_provider_client`.
- Don't hard-code a provider base URL outside `providers.PROVIDERS`.
- Don't auto-derive model tags except via the catalog's curated allow-list (`long_context` / `cheap` / `fast` / `multimodal`); the opinionated `coding` / `reasoning` / `frontier` tags in `MODEL_METADATA` are curated-only.
- Don't generate an auto-description as a floor — if no real description is available, the model is hidden, not faked.
- Don't add a "show all uncatalogued models" escape hatch to the picker without taking it through a design review — the strict gate is the user-trust contract.
- Don't read or write `~/.wb_coding_agent/{model_catalog,openrouter_catalog}.json` outside `model_catalog.py`.
- Don't introduce a third-party metadata source as a fallback for missing curated pricing — the contract is "curated, OR (for OpenRouter) OpenRouter's own catalog, OR hidden". Pricing must come from a source the user is actually paying (the provider directly, via curation; OpenRouter directly, via their own API for the OpenRouter namespace). Anything else silently widens the integrity surface.
- Don't write media bytes (image / audio / video / embeddings) or attachment bytes to disk outside the `_artifact_paths.py`-owned folder structure.
- Don't add new media-mode pricing fields to `ModelInfo`'s top level — extend the existing per-mode slots so the per-mode strict gate keeps applying.
- Don't extract PDF text outside `attachments.extract_text` — `pypdf` is the single import site.
- Don't render a model selectbox anywhere — the modal `_model_picker_dialog` is the single model-selection UI.
- Don't construct `openai.OpenAI` for user-facing chat completions — that path goes through `providers.make_provider_client(...)` (which builds the client at connect time) and then `chat_streams.stream_chat` (which uses it). The catalog-fetch path inside `model_catalog.py` reads through the same stored client; nothing outside those two layers should construct an OpenAI client directly.
- Don't bypass `chat_streams.stream_chat` to get at provider-native streaming — extend the three-way dispatch instead.
- Don't enrich a non-OpenRouter provider's models from OpenRouter's catalog (pricing, descriptions, capabilities) — the OpenRouter catalog enriches `openrouter:*` ids only. Cross-provider enrichment silently substitutes one provider's framing for another's offering.

## How a turn works

A "turn" is one user prompt → agent reply for the **active chat**. The chat
data model + threading harness lives in [`chats.py`](chats.py); the chat page
([`app_pages/chat.py`](app_pages/chat.py)) is purely a renderer driven by an
`@st.fragment(run_every="0.25s")` that polls the active chat while a turn is
in flight.

1. The chat page calls `_start_turn(prompt, override_model=None)`, which mirrors the user's current dropdown picks (model / mode / working_dir) onto `chat = ss.chats[ss.active_chat_id]`, runs `account.apply_git_identity` once per `(working_dir, github login)` pair, and delegates to `chats.start_turn(chat, prompt, ss.client, override_model=...)`.
2. `chats.start_turn` synchronously seeds the user message + assistant placeholder under the chat's lock (so the next render sees them immediately), flips `chat.status` to `"running"`, persists, and spawns a daemon thread that drives the agent loop.
3. The background thread iterates `agent.run_agent_turn(client, model, chat.messages, working_dir, mode)`. `mode` is either `"agent"` (full tool set, default) or `"ask"` (read-only tools and an ask-only system prompt). The agent runs `project_context.scan(working_dir)` and then `select_skills_for_turn(last_user_message, ctx)`, yields a single `skills_loaded` event followed (when Weave is initialized) by a `weave_trace` event carrying this turn's `Call.ui_url`, and rewrites `chat.messages[0]` with the base system prompt + the project-context addendum (AGENTS.md / CLAUDE.md / .cursor/rules eagerly + the matched skills' bodies). Mid-conversation mode switches and edits to AGENTS.md take effect on the next turn for free.
4. The agent loop sends `messages + tools=tools_for_mode(mode) + mcp_servers.get_registry().openai_tool_schemas() + tool_choice="auto" + stream=True + stream_options={"include_usage": True}` to W&B Inference. (MCP tools are only included in Agent mode; Ask mode's contract stays "purely sandbox-read".) The streaming response is consumed by `_stream_one_call`, which yields `assistant_text_delta` events for each `delta.content` chunk, accumulates `delta.tool_calls` fragments by `index`, and captures the trailing empty-`choices` chunk that carries `usage.prompt_tokens` / `usage.completion_tokens` / `usage.total_tokens`.
5. The background thread accumulates `assistant_text_delta` chunks into `chat.partial_text` (under the chat's lock) and appends every other event (`skills_loaded`, `weave_trace`, `assistant_text`, `tool_call`, `tool_result`, `error`, `weave_pruned`) to the in-flight assistant turn's `events` list (also under the lock). It persists to disk after every event so a crash mid-turn loses at most one event.
6. While there are tool calls, the outer agent loop dispatches each through `mcp_servers.dispatch` when the name starts with `mcp__` and `tools.dispatch` otherwise, appends a `{"role": "tool", "tool_call_id": ..., "content": json.dumps(result)}` message, and re-queries (back to step 4 with another streamed call). When a streamed response contains no tool calls the loop yields the final `assistant_text` event and breaks out.
7. After the tool loop ends, `run_agent_turn` calls `wb_client.prune_oldest_calls_if_pressured()` and (only when Weave reported storage pressure during the turn) yields a `weave_pruned` event. The background thread also records per-turn token usage via `usage.record_usage(...)` and appends a synthesized `turn_usage` event for replay.
8. If `chat.title` is still the default `"New chat"` and the turn produced an assistant message, the background thread asks DeepSeek for a 5-word title via `chats.generate_title` (which calls `agent.generate_text` with `agent.DEEPSEEK_MODEL`); on any failure it falls back to `chats.derive_title` (a truncation of the first user message). The title is set once and never refreshed; a previous iteration of the turn loop also asked DeepSeek for a one-sentence description on every turn but that was removed when the chat-list redesign deleted the only UI surface that read it.
9. The thread flips `chat.status` to `"idle"` (or `"error"` with `error_message` on exception), clears `partial_text`, and persists.

While the background thread runs, the chat page's `_render_active_chat_live(chat.id)` fragment polls every 0.25s, snapshots `chat.ui_turns` + `chat.partial_text` + `chat.status` under the lock, and re-renders. When `status` flips out of `"running"`, the fragment triggers a top-level `st.rerun()` so the static renderer (`_render_active_chat_static`) takes over and the poll stops.

`skills_loaded`, `weave_trace`, `assistant_text`, `tool_call`, `tool_result`, `error`, `weave_pruned`, and `turn_usage` events are persisted to the chat's `ui_turns` for replay; `assistant_text_delta` events are display-only (the trailing `assistant_text` carries the full content for replay), and the per-round `usage` events are aggregated into a single `turn_usage` event by the background runner rather than persisted individually (the dashboard reads from `usage.jsonl`, not from the in-session log).

### Usage capture

After the inference loop finishes (regardless of whether it ended via final answer or error mid-turn), `app_pages/chat.py._run_turn` builds a single per-turn record via `usage.build_entry(...)` and writes it to `~/.wb_coding_agent/usage.jsonl` via `usage.record_usage(...)`. The schema is documented in `usage.py`'s module docstring; bumping it requires editing both that docstring and `build_entry`. The same record's tokens + cost are mirrored into a `turn_usage` UI event appended to the assistant turn's events list and rendered as a compact caption (`:material/data_usage: 13.2k tokens · $0.014 · 3 rounds · DeepSeek V3.1 · :material/sensors: View trace`). The model label is sourced from `models.model_label(...)` so a mid-chat model switch is visible on every past turn (not just the model-card line below the chat input, which only reflects the *current* selection); same-turn pricing changes from a swap are immediately attributable. The trace link is folded into that caption when the per-turn `weave_trace` event recorded a URL; when there is a `weave_trace` event but no `turn_usage` (e.g. the turn errored before any tokens were used), the chat renderer falls back to a standalone `:material/sensors: View trace in Weave` caption so the link is never dropped. The Usage page (`app_pages/usage.py`) reads the on-disk log on every render, so usage shows up across browser sessions and across `streamlit run` invocations. Cost is computed by `usage.compute_cost`, which reads pricing fields from `models.MODEL_METADATA` and returns `None` for models without published pricing — the dashboard renders such turns with `-` in the cost cell rather than zeroing them silently.

Each `Chat` carries its own OpenAI-format conversation (`chat.messages`, used for the API) **and** its own structured render log (`chat.ui_turns`, used for replay). Keep them parallel — if you add a new event type, replay must understand it too. Both live entirely inside [`chats.py`](chats.py); no other module mutates them.

Concurrently with steps 3–7, every call is captured in W&B Weave as a single nested trace: `run_agent_turn` (kind=`agent`, purple) is the parent op; each invocation of `_stream_one_call` (kind=`llm`, blue) is a child, with the auto-patched `client.chat.completions.create` underneath; each `tools.dispatch` invocation (kind=`tool`, green) is a sibling of the inference calls. The `client` argument is stripped from logged inputs by `_strip_client` so the W&B API key is never serialized into a trace. See "Tracing with W&B Weave" below for the contract.

## Tracing with W&B Weave

`wb_client.init_weave(api_key, project)` is called by `streamlit_app.py._connect` after a successful Connect. It:

1. Sets `WANDB_API_KEY` in the process environment (so `weave.init` does not block on stdin auth — important inside the packaged desktop app where there is no terminal).
2. Calls `weave.init(project or DEFAULT_WEAVE_PROJECT)`, which patches `openai.OpenAI` so every `chat.completions.create` becomes a child of any active `@weave.op`.
3. Returns `(WeaveClient, resolved_project)` so the UI can show `Tracing turns to W&B Weave at <project>` in the sidebar.

`DEFAULT_WEAVE_PROJECT = "wandb-coding-agent"` (in `wb_client.py`) is the fallback project used when the user leaves the optional `team/project` field blank; Weave creates it under the user's default entity on first call.

If init fails (network error, project access, etc.) the UI shows `Weave tracing disabled: <error>` instead. The `@weave.op` decorators in `agent.py` and `tools.py` are no-ops without an initialized Weave client, so failure is non-fatal — the agent runs unchanged, just without tracing.

When you add a new function that calls the W&B Inference service (or that should appear in the trace tree), decorate it with `@_op(...)` (the local compat wrapper — see "weave.op compat shim" below) using one of the standard `kind` values:

- `kind="agent"` — top-level orchestration of a user turn.
- `kind="llm"` — a function whose body issues `client.chat.completions.create` (or another model call).
- `kind="tool"` — tool dispatch / executor.
- `kind="search"` — retrieval helpers, when you eventually add them.

If your decorated function takes an OpenAI client (or anything else with embedded credentials) as an argument, also pass `postprocess_inputs=_strip_client` (or an equivalent function) so secrets stay out of the trace. The auto-patched child OpenAI op already captures every relevant request field (model, messages, tools), so dropping the client argument from the parent's logged inputs costs nothing.

### weave.op compat shim

`agent.py`, `tools.py`, and `mcp_servers.py` each define a tiny local `_op(...)` wrapper that delegates to `weave.op` after stripping `kind=` / `color=` kwargs that older `weave` (< 0.52.24) doesn't accept. The kwargs were added partway through the 0.52 series, and on older installs `@weave.op(..., kind="tool", color="green")` raises `TypeError` at decorator-evaluation time — i.e. at module import — which crashes the whole app before Streamlit ever paints. Feature-detecting once at module load (via `inspect.signature(weave.op).parameters`) means upgrade-laggers still get a correct trace tree; they just lose the Weave UI's color/category badges. `pyproject.toml` pins `weave>=0.52.24` so fresh installs get the full UX, but the shim is the safety net.

When adding a new `@_op`-decorated function, always go through `_op` — never call `@weave.op(..., kind=..., color=...)` directly. If a future weave release rejects yet another kwarg, extend `_WEAVE_OP_DROP` in each of the three modules to include it.

## Adding a new tool (the dual-edit pattern)

Tools touch filesystem and shell, so getting them right matters. Follow these steps in order:

1. **Add the schema** in `TOOL_SCHEMAS` in `tools.py`. Use OpenAI function-calling format. Required fields belong in `parameters.required`. Write a description that tells the model exactly when to use the tool and what it returns.
2. **Implement the executor** as `_my_tool(ctx: ToolContext, ...) -> dict[str, Any]`. Always:
   - Call `_resolve_inside(ctx.working_dir, path)` for any path argument.
   - Return `{"error": ...}` for recoverable failures so the model can adjust. Do not raise.
   - Keep the result JSON-serializable - it is round-tripped through `json.dumps` for the next API call.
3. **Register it** in the `_DISPATCH` map in `tools.py`.
4. **Pick a UI icon** in `TOOL_ICONS` in `streamlit_app.py` (Material icon name).
5. **Render its result** in `_render_tool_event` in `streamlit_app.py` if it has a non-trivial display (diffs, listings, stdout, etc).
6. **Update the system prompt** in `agent.py` (`SYSTEM_PROMPT`) if the tool changes the recommended workflow.
7. **Update the README and this file's tool table** below if the tool is user-visible.

Current tools:

| Tool | Required | Returns |
| --- | --- | --- |
| `list_files(path=".", max_depth=3)` | - | `{"listing": str}` (tree) |
| `read_file(path, start_line=1, end_line=None)` | `path` | `{"content": str, "total_lines": int, ...}` (line-numbered) |
| `write_file(path, content)` | `path`, `content` | `{"ok": True, "diff": str, ...}` (unified diff vs prior) |
| `edit_file(path, old_string, new_string)` | all | `{"ok": True, "diff": str}` after a unique replace |
| `run_shell(command, timeout=30)` | `command` | `{"exit_code", "stdout", "stderr"}`. Runs with `cwd=working_dir`; no in-product gate. |

In **Ask only** mode, only `list_files` and `read_file` are exposed to the model — `write_file`, `edit_file`, and `run_shell` are withheld at the API boundary by `tools.tools_for_mode("ask")`. The `READONLY_TOOL_NAMES` set in `tools.py` is the source of truth for which tools remain available in Ask mode; update it (and the system prompt for Ask mode in `agent.py`) if you add a new read-only tool. **MCP-backed tools are also withheld in Ask mode for v1**: `agent.py` only adds `mcp_servers.get_registry().openai_tool_schemas()` to the tools list when `mode != "ask"`, because we don't yet have per-MCP-tool read-only metadata.

### MCP-backed tools

External MCP server tools are exposed to the model under `mcp__<server_id>__<tool_name>` names. They never enter `TOOL_SCHEMAS`. The dual-edit pattern above does **not** apply — adding an MCP tool is purely a user-side action (pointing the agent at the right MCP server), not a code change. To make a new MCP server type easier to use, edit `_render_mcp_result` in `streamlit_app.py` if its result content blocks need a richer rendering than the default text/image/resource walk.

## Skill selection contract

`project_context.py` is the single source of truth for what skills look like to the agent — discovered from any directory listed in `SKILL_ROOT_SUBPATHS` (today: `.cursor/skills/` and `.claude/skills/`, scanned at both workspace and user scope). Every turn, `agent.py` calls `project_context.scan(working_dir)` and then `select_skills_for_turn(last_user_message, ctx)`. The selection rule is:

1. **Slash commands** — any `/<slug>` token in the user's message (matched at word boundaries via `_SLASH_RE` in `project_context.py`) force-loads that skill for the turn. Slash commands always win and are not subject to the per-turn cap.
2. **Keyword match** — for every skill, we mine trigger phrases from (a) the YAML frontmatter `triggers:` field, (b) any `Triggers: a, b, c` sentence inside the frontmatter `description`, and (c) word-parts of the skill's `name` ≥ 3 characters. A skill matches if any phrase appears in the user message (whole-word for word-character phrases, substring for phrases containing punctuation like `st.`).
3. Keyword matches fill the remaining budget up to `MAX_AUTO_SKILLS` (currently 5), ordered by trigger-match count.
4. Each loaded SKILL.md body is capped at `SKILL_CONTENT_CAP` characters (currently 6000).

**Authoring a new SKILL.md.** Frontmatter must include `name:` (used as the slash slug after slugification). To make the skill reliably keyword-trigger, pack a `Triggers: a, b, c` line into `description` — this matches the convention the existing Cursor skills already use. Write triggers as the words / phrases users naturally type (e.g. `dashboard, metrics, KPI`) rather than the skill's internal terminology.

**Workspace vs. user skills.** `SKILL.md` files under any directory in `SKILL_ROOT_SUBPATHS` — today `.cursor/skills/` and `.claude/skills/` — feed into the same selection pass at both workspace (`<wd>/<sub>/**/SKILL.md`) and user (`~/<sub>/**/SKILL.md`) scope. Within a scope, slug collisions fall through to the first directory in `SKILL_ROOT_SUBPATHS` order (so a project that has both `.cursor/skills/foo` and `.claude/skills/foo` keeps the `.cursor/skills/foo` entry). Across scopes, the workspace skill wins (project context overrides global) and the conflict is recorded in `summary(...)["slug_conflicts"]` so the UI can warn.

## Coding standards

All future code additions must:

- **Be commented and documented.** Every module needs a top-of-file docstring explaining its responsibility. Every public function needs a docstring describing arguments, return shape, and any side effects. Add inline comments for non-obvious branches, safety boundaries, and protocol decisions - **do not** add narrative comments that just restate the code (`# increment counter`).
- **Be typed.** Use `from __future__ import annotations` and PEP-604 (`X | None`) types. Public function signatures must have full annotations.
- **Stay inside the module map above.** If a new responsibility genuinely does not fit, update both the code and the table in this file.
- **Not introduce a new dependency** without adding it to `[project].dependencies` in [`pyproject.toml`](pyproject.toml) with a sensible lower bound (`>=`). Prefer the OpenAI SDK and the standard library; reach for new packages only when they materially simplify the code.
- **Keep paths relative to the working directory** when interacting with the user's project. Never widen the sandbox.
- **Show users what the agent did.** Tool results that mutate state must include a `diff` (or equivalent before/after evidence) so `_render_tool_event` can display it.
- **Keep the in-app Docs page in sync, and keep its voice.** Whenever you add, remove, or rename a user-facing surface — a tab, a setting, a button label, a tool, a sidebar panel, a chart, a slash-command behavior, a popover, a dialog, a dropdown option, or anything else the end user sees — update [`app_pages/docs.py`](app_pages/docs.py) in the same change. New copy on that page is plain English at roughly an 8th-grade reading level; define every jargon term (W&B Inference, agent, token, trace, MCP, branch, commit, push, rebase, pull request, working directory, PAT, etc.) inline at first use, in short sentences. The Docs page is the user-visible companion to this file.

Streamlit-specific rules:

- Read [`/Users/john/streamlit-inference/.cursor/skills/developing-with-streamlit/SKILL.md`](.cursor/skills/developing-with-streamlit/SKILL.md) and the relevant sub-skill before non-trivial UI changes.
- Use Material icons (`:material/icon_name:`) over emojis.
- Use sentence case for titles and labels.
- Prefer `st.caption` / `st.toast` over `st.info` for low-importance notes.
- Pin `streamlit>=1.56.0`. Many features used here (Material icons, `st.space`, `horizontal_alignment`, modern `st.container` options, the `on_dismiss=` parameter on `@st.dialog` — see the dialog rule below) require this.
- **Every `@st.dialog` whose visibility is gated by an `ss.<flag>` boolean / id MUST pass `on_dismiss=<close_callback>`.** Streamlit closes the dialog visually when the user clicks the built-in **X**, presses **Esc**, or clicks outside, but those built-in dismissal paths do **not** fire the dialog's body Cancel / Save handlers — so the gating flag stays stuck at "open" and the dialog re-mounts on the very next rerun (e.g. when the user presses Enter in `st.chat_input` to send a message). The `on_dismiss` callback is the only hook Streamlit gives you for those built-in dismissal events; wire it to a tiny helper that just clears the flag(s) (e.g. `_close_diff_dialog`, `_close_new_project_dialog`, `_close_new_branch_dialog`, `_close_first_push_dialog`, `_close_mcp_dialog`, `_close_delete_chat_dialog`). Define the helper **above** the `@st.dialog`-decorated function so the decorator can reference it at module-load time. Do not skip this on dialogs that "shouldn't" be dismissable in practice — users dismiss everything.

## Anti-duplication checklist

Before creating any new file or function, verify:

- [ ] Does this responsibility already live somewhere in the **module map** above? If yes, edit there.
- [ ] Am I about to write a second OpenAI client constructor? Use `wb_client.make_client`.
- [ ] Am I about to redefine a tool schema? It belongs in `TOOL_SCHEMAS` in `tools.py`.
- [ ] Am I hard-coding the W&B Inference base URL? Import `WB_INFERENCE_BASE_URL` from `wb_client`.
- [ ] Am I about to read or write the user's filesystem outside `tools.py`? Stop. Add a tool instead.
- [ ] Am I about to call `client.chat.completions.create` outside `agent.py`? Stop. Extend `run_agent_turn` (for streaming, tool-calling turns) or `generate_text` (for one-shot, non-streamed completions like commit messages or PR titles).
- [ ] Am I shelling out to `git` (`subprocess.run(["git", ...])` / `os.system("git ...")`) anywhere other than [`git_ops.py`](git_ops.py)? Stop. Add a helper to `git_ops.py` (with a `GitError`-raising wrapper) and call it from there. The narrow grandfathered exception is `account.py`'s project-bootstrap helpers (`git_init`, `git_clone`, `git_add_remote`, `apply_git_identity`); do not extend that list.
- [ ] Am I building a `https://github.com/.../compare/...` (or GitLab merge-request, or Bitbucket pull-request) URL by hand? Stop. Use `git_ops.remote_compare_url(working_dir, branch, base, *, title, body)` so the per-host URL templates and URL-encoding stay in one place.
- [ ] Am I hard-coding a DeepSeek model id for AI-driven git text (commit messages, PR titles, conflict resolution)? Use `agent.DEEPSEEK_MODEL`. The chat-page push pipeline (via `commit_ai.py`) and the merge-conflict resolution turn both pull from that single constant; mismatched ids would silently bypass the `commit_ai.is_deepseek_available` check the chat-page Push button runs before showing the AI affordance.
- [ ] Am I adding a new function that calls a W&B Inference model? Decorate it with the local `@_op(...)` shim (`kind="llm"` or `kind="agent"`) and use `postprocess_inputs=_strip_client` if it takes the OpenAI client as an arg. See "Tracing with W&B Weave" + "weave.op compat shim".
- [ ] Am I adding a `@weave.op(..., kind=..., color=...)` decorator directly? Stop. Use the local `@_op(...)` shim instead — older `weave` (< 0.52.24) raises `TypeError` at import time on those kwargs. The shim feature-detects and gracefully drops them.
- [ ] Am I about to call `weave.init` outside `wb_client.init_weave`? Stop. Add it once at Connect; everything else relies on the auto-patched OpenAI client and `@weave.op`.
- [ ] Am I about to attach a second `logging.Handler` to the `weave` logger tree, or write a separate "is the user low on Weave storage?" check? Stop. The `_StorageHandler` in `wb_client.py` is the single listener and `prune_oldest_calls_if_pressured()` is the single reactor. Extend those instead of adding a parallel mechanism.
- [ ] Am I about to call `client.server.calls_delete` or otherwise delete Weave traces from outside `wb_client.py`? Stop. Use `prune_oldest_calls(...)` so the 1000-call cap, root-only filter, and error swallowing stay in one place.
- [ ] Am I about to redefine model display labels, descriptions, **or pricing**? They live only in `MODEL_METADATA` in [`models.py`](models.py), transcribed from the W&B [Available models](https://docs.wandb.ai/inference/models) page and the [Inference pricing](https://wandb.ai/site/pricing/inference) page. Edit there. `app_pages/chat.py`, `app_pages/usage.py`, and `usage.py` all read from this dict.
- [ ] Am I about to compute USD cost in any module other than [`usage.py`](usage.py)? Stop. `usage.compute_cost(model, prompt, completion)` is the single price-times-tokens calculator; the dashboard, the per-turn footer caption, and the on-disk log all flow through it.
- [ ] Am I about to read or write `~/.wb_coding_agent/usage.jsonl` outside [`usage.py`](usage.py)? Stop. Use `usage.record_usage` / `usage.load_usage` so the schema stays in one place.
- [ ] Am I about to read or write `~/.wb_coding_agent/credentials.json` or `preferences.json` outside [`account.py`](account.py)? Stop. Those files are mode-0600 secrets (`credentials.json`) or non-secret prefs that other code shouldn't touch; extend `account.py` instead.

Adding a "save on every keystroke without debounce" path defeats the 200ms debounce that was sized to keep websocket traffic tiny without leaving the on-disk draft persistently stale. ## Desktop build

The repo ships a packaged desktop app on top of [`streamlit-desktop-app`](https://pypi.org/project/streamlit-desktop-app/). The build pipeline is owned by [`scripts/build_desktop.py`](scripts/build_desktop.py); do not invoke `streamlit-desktop-app build` directly. Architecture at runtime:

- The packaged binary spawns Streamlit on a random localhost port (headless).
- `pywebview` opens a native window pointed at that URL (WKWebView on macOS).
- The user-visible window is therefore the same Streamlit UI, with the developer-mode toolbar tooling suppressed via `client.toolbarMode = "viewer"` (the Settings menu — used for the built-in light/dark/system theme toggle — stays reachable; "minimal" would hide it too and there's no programmatic theme switch in Streamlit).

Build prerequisites:

- Use a CPython 3.12 interpreter; `streamlit-desktop-app` 0.3.4 caps Python at 3.12.
- Install the build extras into a separate venv (e.g. `uv venv --python 3.12 .venv-build && uv pip install -e '.[desktop]'`).

Build invocation (from repo root, with the build venv active):

```bash
python scripts/build_desktop.py
```

Or use the bootstrap wrapper, which works from a clean shell — no venv activation required, and it (re-)syncs the build venv against `pyproject.toml` first so the build can never run with stale deps:

```bash
./scripts/build_desktop.sh
```

Output:

- macOS: `dist/WB Coding Agent.app` (real `.app` bundle; drag to `/Applications`). Unsigned, so first launch needs right-click → Open or `xattr -d com.apple.quarantine`.
- Linux / Windows: `dist/WB Coding Agent/` (PyInstaller onedir layout). Distribute as a zip.

When adding a new top-level Python module imported by `streamlit_app.py`, append it to `BUNDLED_MODULES` in `scripts/build_desktop.py`. When adding a new third-party dependency that is only imported transitively from a bundled module, append it to `COLLECT_ALL_PACKAGES`. PyInstaller's static import graph cannot see imports inside files added via `--add-data`, so this manual step is unavoidable.

When changing any Streamlit runtime option, update **both** `.streamlit/config.toml` (for local dev) **and** `STREAMLIT_OPTIONS` in `scripts/build_desktop.py` (for packaged builds). The two are not auto-synced because the bundled app cannot read the config file at runtime.

Out of scope for the desktop build today: code signing, notarization, auto-update. The artifact is intended for the user's own machine; if you ship it externally, expect Gatekeeper friction.

## Out of scope (intentional)

These are explicitly **not** part of v1. If a future request asks for one of these, treat it as a new feature, design it deliberately, and update this section once it lands:

- Per-tool click-to-approve UI, and any in-product gating of file writes or shell commands. Current safety model: the user picks a working directory they trust the agent to operate inside; file ops auto-approve and show diffs; shell commands run with that directory as their cwd and a 30s timeout. There is no toggle, allow-list, or confirm dialog.
- Manual chat rename. Titles are AI-generated once after the first turn (via `chats.generate_title` calling `agent.DEEPSEEK_MODEL`) and never edited again.
- Per-chat usage attribution in the Usage dashboard. Per-turn rows in `usage.jsonl` aren't tied to a chat id; if a user wants to see "tokens spent in chat X" we'd need to add a `chat_id` field to the schema.
- Streaming for **non-active** chats. Background turns on inactive chats still accumulate `ui_turns` events under their lock, but `partial_text` is rendered only for the active chat in the chat-page fragment.
- Retry for an in-flight turn — re-sending the prompt is still a manual step.
- An auto-resume affordance for a turn the user stopped — once the user clicks **Stop**, any partial reply is preserved in the chat history (with a "Stopped by user" marker) and the conversation history closes out cleanly with a stub assistant message, but resuming where the model left off would require sending the partial back as the assistant's prior turn and asking it to continue. Today the user just sends a new prompt instead. (A chat persisted with `status == "running"` at app startup is still rewritten to `"error"` so a crash mid-turn doesn't leave a phantom spinner.)
- Code signing, notarization, and auto-update for the desktop build (see "Desktop build" above for the current unsigned-distribution caveats).
- Hosted multi-tenant deployment (Streamlit Community Cloud, etc.). The agent's tools mutate the working directory; on a hosted server that directory is shared between visitors. Designing this safely would require per-session sandboxes and is intentionally not v1.
- Git push / PR features intentionally **not** in v1: `gh` (or `glab`) CLI integration; force-push or any other history-rewriting operation; in-product branch creation/deletion; non-rebase merge strategies (the push flow always pulls with `--rebase`); GPG-signed commits (push runs with `commit.gpgsign=false` to avoid passphrase deadlocks in the desktop app); conflict resolution by any model other than `agent.DEEPSEEK_MODEL`; remote-compare-URL templates for self-hosted Git platforms (GitLab/Gitea/Bitbucket Server) — those fall through to surfacing the raw "Create a pull request..." line from `git push` stderr instead.
